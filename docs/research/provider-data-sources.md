# Provider data-source viability

**Decision:** treat Codex and OpenCode Go as separate commercial products, and expose only evidence-backed values. A technical API rate limit must never be presented as a customer subscription quota.

**Research access date:** 2026-07-14. This report used public documentation only; no account, local authentication material, or provider request was accessed.

## Vocabulary and classification

| Term | Meaning |
|---|---|
| Commercial quota | An allowance associated with a customer plan or credit purchase. |
| Technical rate limit | A service-throttling limit such as requests or token throughput within a time window. It is not a commercial quota. |
| Token use | Consumption measured in model-processing units; it may be converted to credits or affect a technical throughput limit. It is not necessarily a plan allowance. |
| Balance | A purchasable or granted credit amount. It is distinct from an allowance and may not be publicly queryable. |
| Used/remaining percentage | A derived ratio only when the same source supplies compatible used and total values. |
| Reset timestamp | A time when an identified rate-limit or allowance window becomes available again; it must retain the source window's semantics. |

**Availability classification:** `supported` (public evidence identifies an observable source), `unavailable` (no public source establishes it), `unknown` (the product may expose it, but public evidence is insufficient), or `requires more research` (a prospective path needs evidence before implementation).

**Conclusion vocabulary:** `viable`, `conditional`, `unavailable`, `requires more research`. `conditional` means a value is publicly documented but requires a user-authorized, authenticated interaction in a future phase.

## Provider identity resolution

OpenCode's official provider documentation identifies **OpenCode Go** as a low-cost subscription plan provided by the OpenCode team, not the Go programming language or an independently verified API product. Its documentation states that a user signs in, adds billing details, and supplies an API key through OpenCode's connection flow. The same page describes OpenCode as an agent that supports many upstream LLM providers, so upstream-provider data must not be attributed to OpenCode Go. [OpenCode providers documentation](https://opencode.ai/docs/providers/#opencode-go) (accessed 2026-07-14).

## Implementation boundary

This research authorizes no provider implementation. A later, separately approved discovery phase may inspect public product documentation again and design a user-authorized source only for entries classified `conditional` or `requires more research`. Entries classified `unavailable` remain absent rather than synthesized.
